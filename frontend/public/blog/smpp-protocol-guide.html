SMPP Protocol Explained: Complete Enterprise SMS Integration Guide

📅 May 2025⏱ 15 min read🏷 SMPP · Enterprise · SMS Integration

SMPP is the backbone of enterprise bulk SMS — used by carriers, aggregators, and large enterprises worldwide. This guide explains SMPP v3.4 from PDU structure to throughput optimization, without the jargon.

What Is SMPP and When Do You Need It?

SMPP (Short Message Peer-to-Peer) is a protocol that creates a persistent TCP connection between your messaging system and an SMS gateway. Unlike HTTP/REST APIs that create a new connection for each request, SMPP maintains an always-on session, enabling throughputs of thousands of messages per second.

Use SMPP WhenUse REST API When
Sending >100,000 messages/daySending <50,000 messages/day
Need <50ms per-message latencyStandard 100–300ms latency acceptable
Running a telco or aggregatorBuilding a business application
Need asynchronous DLR deliveryWebhook-based DLR acceptable
Have experienced telecom engineersStandard web development team

SMPP Architecture

An SMPP connection involves two roles:

  • ESME (External Short Message Entity): Your application — the client that connects to the SMSC
  • SMSC (Short Message Service Centre): The gateway server — Kaolach's infrastructure in this case

The connection is established via TCP (typically port 2775 or 2776 for TLS). Once connected and authenticated (bind), messages are exchanged as PDUs (Protocol Data Units) in both directions simultaneously.

SMPP Bind Types

Bind TypeDirectionUse Case
bind_transmitterSend onlyHigh-volume outbound bulk campaigns
bind_receiverReceive onlyDLR reception, inbound SMS handling
bind_transceiverBoth directionsStandard two-way messaging

For most enterprise integrations, use bind_transceiver — it allows you to send messages and receive DLRs on the same connection.

PDU Structure

Every SMPP operation is a PDU with a 16-byte header:

Bytes 0-3:   Command Length   (total PDU length including header)
Bytes 4-7:   Command ID       (e.g., 0x00000004 = submit_sm)
Bytes 8-11:  Command Status   (0x00000000 = ESME_ROK = success)
Bytes 12-15: Sequence Number  (increments per request; used to match responses)
Bytes 16+:   Body             (command-specific parameters)

Key PDU Types

PDUCommand IDDirectionPurpose
bind_transceiver0x00000009ESME→SMSCAuthenticate and establish session
submit_sm0x00000004ESME→SMSCSubmit a message for delivery
submit_sm_resp0x80000004SMSC→ESMEAcknowledge submit, return message_id

Encoding for Chinese Characters

Chinese (and other Unicode) characters require data_coding = 0x08 (UCS-2). This halves the characters per segment from 160 to 70. Ensure your SMPP library handles UCS-2 encoding correctly — common source of garbled Chinese SMS.

# Python example using smpplib
import smpplib.client
import smpplib.consts

client = smpplib.client.Client('sip.kaolach.com', 2775)
client.connect()
client.bind_transceiver(
    system_id='your_system_id',
    password='your_password',
    system_type='',
)

# Send Chinese message
message = '您的验证码是 847291，有效期10分钟'
client.send_message(
    source_addr_ton=smpplib.consts.SMPP_TON_ALNUM,
    source_addr='Kaolach',
    dest_addr_ton=smpplib.consts.SMPP_TON_INTL,
    dest_addr_npi=smpplib.consts.SMPP_NPI_ISDN,
    destination_addr='+8613812345678',
    short_message=message.encode('utf-16-be'),  # UCS-2 encoding
    data_coding=0x08,  # Unicode
    registered_delivery=True,
)

Throughput Optimization

Windowing

SMPP supports "windowing" — sending multiple submit_sm PDUs without waiting for each submit_sm_resp before sending the next. Window size determines max in-flight messages:

  • Window size 1: Synchronous (safe but slow: ~5–10 msg/sec)
  • Window size 10: ~50–100 msg/sec
  • Window size 100: ~500–1,000 msg/sec (standard enterprise)
  • Window size 1000: 5,000–10,000 msg/sec (high-volume aggregator)

Your actual throughput limit is set by your account's configured TPS (Transactions Per Second). Contact your gateway to increase TPS for high-volume campaigns.

Multiple Connections

For throughput beyond what a single connection provides, establish multiple bind_transceiver connections (connection pool). Each connection has its own TPS limit; pool them to multiply throughput:

# Example: 10 connections × 100 TPS each = 1,000 TPS
connections = []
for i in range(10):
    c = create_smpp_connection(host, port, system_id, password)
    connections.append(c)

# Round-robin message distribution
def send(message, phone):
    connection = connections[hash(phone) % len(connections)]
    connection.submit_sm(destination_addr=phone, short_message=message)

Handling DLRs via deliver_sm

DLRs arrive as deliver_sm PDUs on the same connection. Parse the short_message body to extract status:

# DLR message body format (SMSC-dependent, common format):
# id:XXXXXX sub:001 dlvrd:001 submit date:2505011023 done date:2505011023
#   stat:DELIVRD err:000 text:Your ve...

import re

def parse_dlr(short_message: bytes) -> dict:
    text = short_message.decode('latin-1')
    patterns = {
        'message_id': r'id:(\S+)',
        'status': r'stat:(\w+)',
        'error': r'err:(\d+)',
    }
    return {k: re.search(v, text).group(1) for k, v in patterns.items()}
