International SMS Gateway API: Developer's Complete Integration Guide 2025

📅 May 2025⏱ 14 min read🏷 API · SMS · Integration

Whether you're building OTP verification, order notifications, or marketing campaigns, integrating an international SMS gateway requires understanding REST vs SMPP, delivery receipts, error handling, and global routing nuances. This guide covers it all.

REST API vs SMPP: Choosing Your Integration Method

International SMS gateways typically offer two integration protocols. Choosing the right one depends on your volume and latency requirements:

FeatureREST API (HTTP)SMPP
Setup complexityLow — any HTTP clientHigh — requires SMPP library
Latency50–200ms per request5–30ms per message
Throughput50–500 msg/sec500–10,000+ msg/sec
Best forOTP, notifications, <1M msg/dayBulk campaigns, >1M msg/day
ConnectionStateless HTTP requestsPersistent TCP session

For most applications — including OTP verification, transactional alerts, and moderate-volume campaigns — the REST API is the right choice. SMPP is reserved for high-throughput enterprise bulk sending.

Handling Delivery Receipts (DLR)

Delivery receipts tell you whether a message was actually delivered to the handset — not just accepted by the carrier. This is critical for OTP use cases where you need to confirm delivery before timing out.

Webhook Setup

Configure a public HTTPS endpoint on your server to receive DLR callbacks:

POST https://yourapp.com/webhooks/sms-dlr
Content-Type: application/json

{
  "message_id": "msg_8f3a92bc1d4e",
  "status": "delivered",
  "delivered_at": "2025-05-01T10:23:45Z",
  "error_code": null,
  "network": "China Mobile"
}

DLR Status Values and Handling

StatusMeaningRecommended Action
deliveredHandset confirmed receiptMark OTP as sent successfully
failedCarrier rejected or undeliverableRetry on alternate route
undeliveredNumber unreachable (off/no signal)Retry after 60s, then fallback
expiredDelivery window exceededSend new OTP, old one invalid
queuedStill in delivery queueWait (timeout after 90s)

Idempotency

Your webhook may receive the same DLR multiple times (retries on network errors). Always process DLRs idempotently using the message_id as the deduplication key:

# Python example
def handle_dlr(webhook_data):
    msg_id = webhook_data['message_id']
    if sms_log.get(msg_id) == 'delivered':
        return  # Already processed, skip
    sms_log[msg_id] = webhook_data['status']
    # ... business logic

International Routing: Multi-Country Best Practices

E.164 Number Formatting

Always submit phone numbers in E.164 format. This is the international standard and prevents routing ambiguity:

# Correct — E.164 format
+8613812345678   # China
+447911123456    # United Kingdom
+12125551234     # United States

# Wrong — will cause routing errors
008613812345678  # Using 00 prefix instead of +
8613812345678    # Missing + prefix
013812345678     # Local format with leading 0

Number Validation Before Sending

Use a Number Lookup (HLR query) API before sending OTPs to validate that a number is active and identify the carrier. This prevents wasting money on dead numbers and lets you route intelligently:

GET /v1/lookup/+8613812345678

{
  "number": "+8613812345678",
  "valid": true,
  "country": "CN",
  "carrier": "China Mobile",
  "line_type": "mobile",
  "ported": false
}

Error Handling and Retry Logic

HTTP Error Codes

200 OK           — Message accepted
400 Bad Request  — Invalid parameters (check number format, missing fields)
401 Unauthorized — Invalid API key
402 Payment Req. — Insufficient balance
422 Unprocessable — Number invalid or carrier rejected
429 Too Many Req — Rate limit exceeded (implement exponential backoff)
500 Server Error  — Gateway error (retry with backoff)

Exponential Backoff for Retries

# Python retry with exponential backoff
import time, random

def send_sms_with_retry(to, text, max_retries=3):
    for attempt in range(max_retries):
        response = gateway.send(to=to, text=text)
        if response.status_code == 200:
            return response
        if response.status_code in [400, 401, 402]:
            raise Exception(f"Non-retryable error: {response.status_code}")
        # Retryable error (429, 500): exponential backoff
        wait = (2 ** attempt) + random.uniform(0, 1)
        time.sleep(wait)
    raise Exception("Max retries exceeded")

Rate Limits and Throughput Planning

Most REST SMS APIs enforce rate limits to prevent abuse. Plan your integration around these limits:

  • Standard accounts: 50–100 messages per second
  • High-volume accounts: 500–1,000 messages per second
  • Batch endpoint: Up to 1,000 recipients per request

If you're sending large campaigns (millions of messages), consider splitting them into waves over several hours rather than attempting to send everything immediately. This also improves deliverability as carriers rate-limit inbound traffic from single sources.

Security Best Practices

  1. Never expose API keys in client-side code — all SMS API calls must come from your backend server.
  2. Validate webhook signatures — DLR webhooks should include an HMAC signature you verify server-side to prevent spoofed callbacks.
  3. Rate-limit OTP requests per user — prevent SMS pumping attacks that can exhaust your balance. Implement: max 3 OTP requests per phone number per hour.
  4. Log all SMS transactions — store message IDs, timestamps, and DLR statuses for audit and debugging.
  5. Monitor spending anomalies — set billing alerts; SMS pumping fraud can spike costs within minutes.
