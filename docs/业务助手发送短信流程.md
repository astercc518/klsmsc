# 业务助手发送短信流程

## 流程概述

客户通过 **Telegram 业务助手** 或 **Web 端** 发送需审核的短信时，完整流程如下（两端同步）：

1. **客户发起短信审核**：输入短信内容，提交审核
2. **自动生成审核记录**：系统创建短信审核单（SmsContentApproval），并转发到该账户绑定通道对应的供应商群
3. **供应商落地审核**：供应商在群内审核，通过或拒绝
4. **通知客户**：审核通过后，系统通知客户
5. **客户点击立即发送**：客户收到通知后，点击「立即发送」按钮，提交发送数据
6. **系统发送**：系统调用 API 完成短信发送
7. **查看发送统计**：客户可在【发送记录】中查看发送信息

## 详细步骤

### 1. 客户发起短信审核

- **Telegram**：在业务助手中点击「短信审核」，**直接发送需审核的文案内容**（无需号码）
- **Web**：在【发送短信】页面输入单条号码和内容，点击「提交审核」；或进入【短信审核】页面提交
- 提交后，系统创建审核记录

### 2. 转发至供应商群

- **供应商群 ID 来源**：优先从客户账号配置的通道对应的供应商获取（资源报价/供应商管理中的「Telegram 群组 ID」），匹配逻辑：
  1. 获取账户绑定的通道（按优先级）
  2. 遍历通道，查找关联供应商的 `telegram_group_id`
  3. 若均未配置，回退到全局 `telegram_admin_group_id`（Bot 配置或 .env）
- 消息仅显示：测试国家、测试内容（不显示账户等用户信息）
- 供应商可点击「✅ 通过」或「❌ 拒绝」，结果会转发给发起客户，每条消息仅可操作一次

### 3. 供应商审核

- **通过**：更新审核状态，向客户发送通知消息，附带「📤 立即发送」按钮
- **拒绝**：更新审核状态，通知客户审核未通过

### 4. 客户点击立即发送

- **Telegram**：客户收到审核通过通知后，点击「📤 立即发送」按钮
  - **只审核文案时**：需再填写接收号码（E.164 格式），然后系统发送
  - **含号码时**：直接调用发送接口
- **Web**：在【短信审核】页面，状态为「已通过」的记录会显示「立即发送」按钮
- 发送成功后，更新审核记录的 `message_id`，并提示客户
- 发送失败时，记录错误信息并提示客户

### 5. 查看发送记录

- 客户可在【发送记录】中查看历史发送详情
- 包含号码、内容、状态、费用等信息

## 配置说明

- **telegram_enable_sms_content_review**：是否启用短信内容审核（默认 true）
- **供应商群 ID**（优先级从高到低）：
  1. **供应商管理**：为通道关联的供应商配置「Telegram 群组 ID」（资源报价/供应商管理 → 编辑供应商）
  2. **telegram_admin_group_id**：全局兜底，Bot 配置或 .env 中的 `TELEGRAM_ADMIN_GROUP_ID`
- 前置条件：账户需绑定通道，通道需关联供应商

## 部署检查

执行部署检查脚本验证配置是否正确：

```bash
./scripts/deploy_check_sms_audit.sh
```

**必填项**：
1. **TELEGRAM_ADMIN_GROUP_ID**（.env）：供应商群 ID
   - 将 Bot 加入目标群
   - 在群内发送任意消息
   - 使用 @userinfobot 或 @getidsbot 获取群 ID（格式如 `-1001234567890`）
   - 在 .env 中设置：`TELEGRAM_ADMIN_GROUP_ID=-1001234567890`

2. **API_BASE_URL**（Docker）：Bot 容器内已默认配置 `http://api:8000`

配置完成后重启 Bot：`docker compose up -d bot`

**诊断接口**：若仍提示「供应商群未配置」或「未绑定」，可用管理员账号调用：
- `GET /api/v1/admin/debug/supplier-group?account_id=17` （按账户 ID）
- `GET /api/v1/admin/debug/supplier-group?tg_id=6650783812` （按 Telegram 用户 ID）
需在 Header 中携带 `Authorization: Bearer <admin_token>`。返回 bound_channels、checked、resolved_group_id 等。

**命令行诊断**（无需 Token）：
```bash
# 替换 6650783812 为你的 Telegram 用户 ID（可用 @userinfobot 获取）
docker compose exec mysql mysql -u smsuser -psmspass123 sms_system -e "
SELECT tb.tg_id, tb.account_id, a.account_name FROM telegram_bindings tb JOIN accounts a ON tb.account_id=a.id WHERE tb.is_active=1;
SELECT ac.account_id, sc.channel_id, s.supplier_name, s.telegram_group_id 
FROM account_channels ac JOIN supplier_channels sc ON sc.channel_id=ac.channel_id 
JOIN suppliers s ON sc.supplier_id=s.id WHERE sc.status='active';
"
```

**API 诊断**（需管理员 Token，从登录后 localStorage.admin_token 获取）：
```bash
ADMIN_TOKEN=你的token ./scripts/call_debug_supplier_group.sh 17
ADMIN_TOKEN=你的token ./scripts/call_debug_supplier_group.sh tg_id=6650783812
```

**手动绑定 TG 到账户**（若提示「未绑定」且诊断显示 TG ID 与库中不一致）：
1. 用户发送任意消息给 Bot，新提示会显示「您的 Telegram ID: xxx」
2. 编辑 `scripts/bind_telegram_to_account.sql` 中的 @tg_id 和 @account_id
3. 执行：`docker compose exec mysql mysql -u smsuser -psmspass123 sms_system < scripts/bind_telegram_to_account.sql`

## 数据模型

- **SmsContentApproval**：短信内容审核表
  - 状态：pending（待审核）、approved（已通过）、rejected（已拒绝）
  - `phone_number` 可为空（只审核文案时），发送时再填写
  - 审核通过后，客户点击「立即发送」才会实际发送
  - 发送结果记录在 `message_id` 或 `send_error` 字段

## 注意事项

- 所有信息以官网 https://www.kaolach.com/ 展示为准
- 审核通过后需客户主动点击「立即发送」才会执行发送，不会自动发送
- 发送记录与 Web 端【发送记录】数据一致
