设计一个工业级、高可用的“正常”短信发送流程，核心思想是**“异步解耦、最终一致性、状态闭环以及异常兜底”**。为了避免前面提到的局部失败、锁争用和并发阻塞问题，我们需要将整个流程拆分为几个清晰的阶段。

以下是一个标准且健壮的短信发送核心架构与流程设计：

1. 接入与接收层 (API Layer) —— 保证高吞吐与快速响应
这一层的核心原则是**“只做最轻量的操作，立刻返回”**。

请求接收：客户端调用 API 提交短信（单发或批次）。

基础校验：API 网关仅做鉴权（AppKey/Token）、参数格式化和基础限流。

落库与入队：

生成全局唯一的 Message_ID（或 Batch_ID）。

将原始请求写入数据库（状态为 PENDING 或 ACCEPTED）。

立刻将任务投递到 MQ（如 RabbitMQ/Kafka 的 sms_receive_queue）。

同步响应：向客户端返回 Message_ID，API 请求结束。

2. 逻辑处理层 (Processing Layer) —— 风控、路由与计费
消费者从 sms_receive_queue 获取任务，进行核心业务逻辑处理。

黑名单与风控校验：检查手机号是否在黑名单、是否触发防骚扰策略（如 1分钟内同一号码最多发1条）、敏感词过滤。如果未通过，直接标记为 FAILED（风控拦截）。

智能路由：根据手机号的运营商（移动/联通/电信）和目标国家，结合通道的成本、质量和当前拥堵情况，选择最合适的下发通道（Channel）。

计费处理（预扣款机制）：

关键点：不要直接扣除余额，而是冻结余额（预扣费）。

因为此时短信还未真正送达，如果后续发送失败是需要退费的。

下发入队（引入事务消息或 Outbox 模式）：

将短信息组装为目标通道所需的格式。

在同一个本地数据库事务中：更新短信状态为 ROUTED（已路由），并将下发指令写入本地 outbox 表或直接使用 MQ 的事务消息推送到 sms_send_queue。这彻底解决了“扣费了但没发出去”的一致性问题。

3. 网关下发层 (Gateway Layer) —— 协议适配与并发控制
专用的网关微服务从 sms_send_queue 消费数据，与真实的运营商/供应商对接。

通道限流：严格按照运营商要求的 QPS 进行令牌桶限流，防止因发送过快被封停账号。

协议转换：将内部的 JSON/Protobuf 转为 SMPP (submit_sm) 或 HTTP 请求。

响应处理：

接收到运营商的同步 ACK 回应（如 SubmitSMResp）。

状态更新：通过 MQ 异步发送一个状态更新事件，将数据库中的状态修改为 SENT（已提交网关）。

4. 回执与结算层 (DLR & Settlement Layer) —— 状态闭环
运营商最终将短信投递到用户手机后，会异步回调我们的系统。

接收回执 (DLR)：网关接收到 deliver_sm（状态报告），不做复杂处理，直接扔进 sms_dlr_queue MQ 中。

批量更新与结算 (Worker)：

状态更新：消费者拉取回执，更新短信最终状态（DELIVERED 送达 或 UNDELIVERED 失败）。注意防锁冲突：不要每条立刻 Update DB，可以将回执结果写入 Redis，由定时任务每 3-5 秒批量 Update 数据库。

资金结算：

如果是 DELIVERED：将之前的冻结金额转为实际扣费。

如果是 UNDELIVERED：将冻结金额执行解冻退费。

回调客户：如果客户配置了 Webhook，在此环节将最终状态推送给客户。

5. 异常兜底层 (Fallback & Cron Jobs) —— 防微杜渐
任何系统都可能丢消息，必须有兜底定时任务（Daemon/Cron）。

超时巡检：每小时扫描一次数据库。如果发现某条短信处于 SENT 状态超过 72 小时（运营商通常保留 72 小时）仍未收到 DLR 回执。

强制结算：将这些僵尸短信状态强制标记为 UNKNOWN（未知）或 EXPIRED（超时），并根据平台协议决定是否给客户退费，确保批次能够正常进入 COMPLETED 状态，不会永远卡住。

通过这种设计：业务逻辑被彻底解耦，高并发不会压垮数据库，异常断电也不会导致客户资金损失或重复发送。
