# 数据导入任务部署说明

## 任务一直显示「等待中」的原因

导入任务依赖 **Celery Worker** 异步处理。任务提交后会进入 RabbitMQ 的 `data_tasks` 队列，由 Worker 消费执行。若 Worker 未启动，任务会一直处于「等待中」。

## 解决方案

### 1. Docker 部署（推荐）

确保 Worker 容器已启动：

```bash
# 启动全部服务（含 Worker）
docker-compose up -d

# 仅启动 Worker
docker-compose up -d worker

# 查看 Worker 日志（确认是否正常消费任务）
docker-compose logs -f worker
```

### 2. 本地开发

需同时启动 API 和 Worker：

```bash
# 终端 1：启动 API
cd backend && uvicorn app.main:app --reload

# 终端 2：通用 Worker（不含 sms_send，避免数据导入占满并发导致短信堆积）
cd backend && celery -A app.workers.celery_app worker --loglevel=info -Q celery,sms_dlr,data_tasks,settlement_tasks

# 终端 3：短信专用 Worker（必开，否则发送记录会长期停在「发送队列中」）
cd backend && celery -A app.workers.celery_app worker --loglevel=info --concurrency=8 -Q sms_send
```

### 3. 前置依赖

Worker 依赖以下服务均已启动：
- **RabbitMQ**（消息队列）
- **Redis**（结果存储）
- **MySQL**（数据库）

### 4. 卡住任务的处理

对「等待中」的任务可点击「重新提交」尝试再次入队。若 Worker 已启动，任务会被立即处理。若长时间仍无进展，请按上述步骤检查 Worker 是否运行。

### 5. 代码变更后需重启 Worker

Celery Worker 在启动时加载 Python 代码，**不会**自动热更新。若修改了 `data_worker.py` 或相关逻辑，必须重启 Worker 才能生效，否则会报 `TypeError: data_import_numbers() takes X positional arguments but Y were given` 等签名不匹配错误。

```bash
# Docker 部署：重启 worker 容器（含短信专用 worker）
docker-compose restart worker worker-sms

# 或重建并启动（代码通过 volume 挂载时通常只需 restart）
docker-compose up -d --force-recreate worker worker-sms
```

## 短信长期停在「发送队列中」（queued）

与通道无关时，常见原因是 **Celery 并发槽被 `data_tasks` 等长任务占满**：短信任务已进 RabbitMQ 的 `sms_send` 队列，但通用 Worker 抽不出进程消费，数据库状态会一直保持 `queued`。

Docker Compose 已拆出 **`worker-sms`** 仅消费 `sms_send`，与数据/结算任务隔离。部署后务必：

```bash
docker compose up -d worker-sms
docker compose logs -f worker-sms
```

若仍不消费：检查 RabbitMQ 管理台 `sms_send` 队列是否堆积、环境变量 `RABBITMQ_*` 是否与 API 一致。
