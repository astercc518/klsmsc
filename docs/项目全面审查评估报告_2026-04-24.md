## 二、🔴 严重问题（应立即修复）

### 2.1 JWT 密钥"隐式自动生成" —— 生产环境定时炸弹

**文件：** `backend/app/config.py:88-93`

```python
@field_validator("JWT_SECRET_KEY", mode="before")
def _ensure_jwt_secret(cls, v):
    if not v:
        return secrets.token_urlsafe(48)  # 静默生成，不报错！
    return v
```

**后果：**
- 生产环境忘配 `JWT_SECRET_KEY` 时不会报错，服务正常启动
- 多副本部署（如 docker-compose scale）时每个 API 实例生成**不同密钥** → token 被随机打到其他实例时 401
- 重启任意实例 → 密钥更换 → **全员强制下线**，故障极难溯源

**修复建议：**
```python
@field_validator("JWT_SECRET_KEY", mode="before")
def _ensure_jwt_secret(cls, v, info):
    env = (info.data or {}).get("APP_ENV", "development")
    if not v:
        if env == "production":
            raise ValueError("生产环境必须显式配置 JWT_SECRET_KEY")
        return secrets.token_urlsafe(48)  # 仅开发环境允许自动生成
    return v
```

同样需要对以下默认密码做相同处理：
- `DATABASE_PASSWORD: str = "smspass"`
- `RABBITMQ_PASSWORD: str = "smsc_mq_pass"`

### 4.2 日志无轮转——磁盘风险


无 `logrotate` 配置，无 Docker logging `max-size`/`max-file` 限制。高流量下可在数天内耗尽磁盘。

**建议（docker-compose.yml 所有服务加 logging）：**

### 4.3 Worker 各自建立独立连接池

`batch_worker.py` 中有 `_worker_engine = create_async_engine(...)` 独立于主 `app.database.engine`。每个 Celery 进程有独立连接池，需核算：

```
DATABASE_POOL_SIZE × (api副本数 + worker进程数) ≤ MySQL max_connections (ProxySQL 上限)
```

目前 ProxySQL 做了连接池但未见读写分离规则说明，建议梳理连接数上限。

### 4.4 `sms_logs.account_id` 类型不一致

`sms_logs.account_id` 定义为 `Integer`，但 `sms_logs.id` 是 `BigInteger`。若 `accounts.id` 将来超过 `Integer` 范围（~21 亿），外键关联会出问题。建议统一为 `BigInteger`。
