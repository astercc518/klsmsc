### P0-1｜修复 JWT 密钥隐式自动生成

**文件：** `backend/app/config.py`  
**风险等级：** 🔴 高危  
**影响：** 生产环境漏配 `JWT_SECRET_KEY` 时服务正常启动但多副本 token 互不认证，重启即全员掉线  
**工作量：** 0.5 小时

#### 修复步骤

**Step 1** 修改 `config.py` validator，生产环境强制要求显式配置：

```python
# backend/app/config.py
@field_validator("JWT_SECRET_KEY", mode="before")
@classmethod
def _ensure_jwt_secret(cls, v: str, info) -> str:
    env = (info.data or {}).get("APP_ENV", "development")
    if not v:
        if env == "production":
            raise ValueError(
                "生产环境必须显式配置 JWT_SECRET_KEY（建议：openssl rand -hex 48）"
            )
        return secrets.token_urlsafe(48)  # 仅开发环境允许自动生成
    return v
```

**Step 2** 对其他敏感默认值做相同处理（写法类似，判断 APP_ENV）：
- `DATABASE_PASSWORD: str = "smspass"` → 生产必填，开发可用默认值
- `RABBITMQ_PASSWORD: str = "smsc_mq_pass"` → 同上

**Step 3** 在 `.env.example` 中补充必填项说明（见 P0-4）

#### 验证方法
```bash

APP_ENV=production JWT_SECRET_KEY="" python -c "from app.config import settings"
# 预期输出：ValueError: 生产环境必须显式配置 JWT_SECRET_KEY

# 验证：开发模式无 key 应能自动生成
APP_ENV=development python -c "from app.config import settings; print(len(settings.JWT_SECRET_KEY) > 0)"
# 预期输出：True
```

### P1-1｜通用仪表盘缓存装饰器

**文件：** `backend/app/utils/cache.py` + 相关统计接口  
**背景：** 仪表盘已修复（Redis 60s 缓存）但同类串行聚合问题在 `/admin/statistics`、`/admin/reports/*` 等接口仍存在  
**工作量：** 3-4 小时

#### Step 1 在 `utils/cache.py` 中实现缓存装饰器

```python
import functools, json, hashlib
from app.utils.cache import get_redis_client

def cached_response(ttl: int = 60, key_prefix: str = ""):
    """
    为 async FastAPI 端点提供 Redis 缓存。
    缓存 key = key_prefix + ":" + sha256(所有参数)
    """
    def decorator(func):
        @functools.wraps(func)
        async def wrapper(*args, **kwargs):
            # 提取 admin/account 对象的 id+role 作为 key 组成
            cache_parts = [key_prefix or func.__name__]
            for v in list(args) + list(kwargs.values()):
                if hasattr(v, "id") and hasattr(v, "role"):
                    cache_parts.append(f"{v.id}:{v.role}")
                elif isinstance(v, (str, int, float, bool)):
                    cache_parts.append(str(v))
            raw_key = ":".join(cache_parts)
            cache_key = hashlib.sha256(raw_key.encode()).hexdigest()[:32]
            full_key = f"cached_resp:{cache_key}"

            try:
                rc = await get_redis_client()
                hit = await rc.get(full_key)
                if hit:
                    return json.loads(hit)
            except Exception:
                pass

            result = await func(*args, **kwargs)

            try:
                rc = await get_redis_client()
                await rc.setex(full_key, ttl, json.dumps(result, default=str))
            except Exception:
                pass

            return result
        return wrapper
    return decorator
```

#### Step 2 应用到以下接口

| 文件 | 端点 | 建议 TTL |
|---|---|---|
| `admin.py` | `GET /admin/statistics` | 60s |
| `admin.py` | `GET /admin/reports/daily-stats` | 120s |
| `reports.py` | `GET /reports/*` | 120s |
| `sms.py` | `GET /sms/stats` 类型接口 | 60s |

```python
"""
启动前自检：验证必要环境变量是否配置、依赖服务是否可达。
在 docker-entrypoint.sh 中 alembic upgrade 之前调用。
"""
import sys, os

REQUIRED_IN_PRODUCTION = [
    "JWT_SECRET_KEY",
    "DATABASE_PASSWORD",
    "RABBITMQ_PASSWORD",
]

def check():
    env = os.getenv("APP_ENV", "development")
    if env != "production":
        return  # 开发环境不做强检

    errors = []
    for key in REQUIRED_IN_PRODUCTION:
        val = os.getenv(key, "")
        if not val or val in ("smspass", "smsc_mq_pass", "请替换", ""):
            errors.append(f"  ✗ {key} 未配置或使用默认值")

    if errors:
        print("❌ 生产环境自检失败，以下必填项未正确配置：")
        for e in errors:
            print(e)
        print("\n请参考 .env.example 完成配置后重启。")
        sys.exit(1)

    print("✅ 生产环境自检通过")

if __name__ == "__main__":
    check()
```

### P2-4｜梳理连接池水位

**背景：** Worker 内部独立建 engine，各 Celery 进程有独立连接池，可能超出 ProxySQL / MySQL 上限  
**工作量：** 1-2 小时

#### 计算当前最大连接数

```
API 副本数 × (DATABASE_POOL_SIZE + DATABASE_MAX_OVERFLOW)
= 1 × (20 + 10) = 30 连接

Worker 进程数 × pool_size_per_worker:
  worker-sms:    concurrency × pool_size
  worker-dlr:    concurrency × pool_size
  worker-result: concurrency × pool_size
  worker-web:    concurrency × pool_size
  beat:          1 进程
```

运行以下命令查看当前实际连接数：
```sql
-- 在 ProxySQL 管理端口 (6032) 执行
SELECT hostgroup, srv_host, ConnUsed, ConnFree, ConnOK, ConnERR 
FROM stats_mysql_connection_pool;
```

#### 建议配置

在 `docker-compose.yml` 中为 Worker 统一设置连接池上限：
```yaml
environment:
  DATABASE_POOL_SIZE: "5"       # Worker 每进程连接数降低
  DATABASE_MAX_OVERFLOW: "3"
```

ProxySQL 设置 `max_connections` 预留 10% 余量。

### P2-5｜`sms_logs.account_id` 类型对齐

**当前：** `sms_logs.account_id = Column(Integer, ...)`，而 `accounts.id = Column(BigInteger, ...)`  
**风险：** 若 account 数量超过 2,147,483,647（约 21 亿）会溢出  
**工作量：** 1-2 小时（含 Alembic 迁移）

新建迁移文件：
```python
# alembic/versions/xxx_fix_sms_logs_account_id_bigint.py
def upgrade():
    op.alter_column("sms_logs", "account_id",
                    existing_type=sa.Integer(),
                    type_=sa.BigInteger(),
                    existing_nullable=False)

def downgrade():
    op.alter_column("sms_logs", "account_id",
                    existing_type=sa.BigInteger(),
                    type_=sa.Integer(),
                    existing_nullable=False)
```
