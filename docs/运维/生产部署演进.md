# 生产部署演进建议

本文描述在 Docker Compose 之上向**更可控的生产形态**演进时的常见方向，便于架构评审与排期（非一次性强制改造）。

## 1. 不可变镜像与覆盖配置

**现状**：`api`、各类 `worker` 常挂载宿主机 `./backend` 源码，利于开发，不利于生产版本可追溯。

**建议**：

- CI 构建 **`backend:标签`** 镜像，`docker-compose` 中 `build` 改为 `image: registry.example/smsc-api:标签`，去掉业务代码 bind mount。  
- 使用 **`docker compose -f docker-compose.yml -f docker-compose.prod.yml`** 覆盖：仅保留日志卷、上传卷、`.env` 注入。

## 2. Celery Beat 高可用

**现状**：全局只能运行 **一个** `beat` 实例；多副本会重复调度。

**建议**：Kubernetes 部署时 Beat 使用 **Deployment replicas=1 + PDB**；若必须多活，需引入 **分布式锁**（如基于 Redis/MySQL 的 ShedLock 模式）或改用 **外部调度器**（如云厂商定时触发 HTTP）触发幂等任务。

## 3. 数据库与 ProxySQL

**现状**：单 MySQL + ProxySQL 连接池。

**建议**：读写分离、主从切换策略与 ProxySQL 规则在**有明确 QPS/延迟指标**后再做；无读写分离时，定期评估 ProxySQL 是否仍带来净收益（少一跳 vs 连接池价值）。

## 4. 密钥与配置

生产应通过 **Secret 管理** 注入 `JWT_SECRET_KEY`、`GRAFANA_ADMIN_PASSWORD`、数据库与 RabbitMQ 密码等，避免写入镜像或明文 compose 仓库副本。

## 5. 可观测性扩展

在现有 Prometheus 抓取 `api`、`worker`、`rabbitmq` 的基础上，可按需增加：

- Celery **Flower** 或 **celery_exporter**；  
- MySQL / Redis **exporter**（`prometheus.yml` 中已预留注释示例）；  
- 日志集中（Loki/ELK）与 `message_id` 关联检索。
