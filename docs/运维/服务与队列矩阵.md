# 服务与队列矩阵（运维）

本文描述 Docker Compose 中各服务职责、Celery 队列与扩容要点，便于排障与变更评审。队列路由以 [`backend/app/workers/celery_app.py`](../../backend/app/workers/celery_app.py) 为准。

## 1. Compose 服务一览

| 服务名 | 主要职责 | 依赖中间件 |
|--------|----------|------------|
| mysql | 主业务库 | 无 |
| proxysql | 应用侧连接池，指向 MySQL | mysql |
| redis | 缓存、Celery result backend | 无 |
| rabbitmq | Celery broker、Go 网关消费 AMQP；**15692** 暴露 Prometheus 插件指标（挂载 `./docker/rabbitmq/enabled_plugins`）。若旧数据卷导致插件未生效，须在低峰评估后重建 `rabbitmq_data` 或手工 `rabbitmq-plugins enable rabbitmq_prometheus` | 无 |
| docker-proxy | 受限 Docker API（仅 CONTAINERS + POST），供 `api` 通过 `DOCKER_PROXY_URL` 做容器级运维；**生产应限制可访问 API 的网络范围**，评估是否改为 CI/CD 操作 | 宿主机 socket |
| api | FastAPI 对外 HTTP、DLR 回调入口等 | proxysql, redis, rabbitmq |
| worker | Celery：`celery`、`sms_dlr`、`data_tasks`、`webhook_tasks`、`integrations` | proxysql, redis, rabbitmq |
| worker-sms | Celery：仅 `sms_send`（HTTP/虚拟主路径；**单条** SMPP 仍可能经此重投 `sms_send_smpp`；**批量 SMPP** 已由 `batch_worker` / API 直投 `sms_send_smpp`） | proxysql, redis, rabbitmq |
| worker-sms-smpp | Celery：`sms_send_smpp`（线程池 Python 路径，**legacy profile**） | proxysql, redis, rabbitmq |
| smpp-gateway | Go：消费 `sms_send_smpp`，多 worker 并行 Submit（每 bind 会话内串行）；环境变量见下文 | proxysql, rabbitmq |
| beat | Celery Beat，**全局须单实例** | proxysql, redis, rabbitmq |
| bot | Telegram 长连机器人 | proxysql, redis |
| landing-preview | Node：`preview-server.mjs`（[astercc518/kaolach](https://github.com/astercc518/kaolach) 快照），供 **frontend** Nginx 将 `kaolach.com` 非控制台路径反代为落地页 | 无 |
| frontend | Nginx 静态资源 + 反代 `landing-preview`（控制台 SPA 仍为 Vue） | landing-preview |
| prometheus / grafana | 监控（compose 默认启动） | prometheus → grafana |
| worker-web | Playwright，`web_automation` 队列 | proxysql, redis, rabbitmq |

## 2. Celery 队列与消费方

| 队列名 | 主要任务（示例） | 默认消费容器 |
|--------|------------------|--------------|
| sms_send | `send_sms_task`（默认路由） | worker-sms |
| sms_send_smpp | `send_sms_task` 重投（SMPP 通道） | **smpp-gateway**（Rabbit 原生消费）；勿与 worker-sms-smpp 同时抢同一队列 |
| sms_dlr | `process_dlr_task`、`process_smpp_dlr_task`、`fetch_dlr_reports_task`、`dlr_timeout_check_task` | worker |
| data_tasks | 数据业务异步任务 | worker |
| celery | `process_batch`、`process_batch_chunk`、`inspect_batches_task`、虚拟 DLR 等 | worker |
| webhook_tasks | `send_webhook`（客户 Webhook 回调） | worker |
| integrations | `okcc_sync_balances_task`（OKCC 等外部集成） | worker |
| web_automation | `web_click_task`、`web_register_task` | worker-web |

## 3. SMPP 部署约定（重要）

- **生产推荐**：启用 **smpp-gateway**，**不要**再启用 **worker-sms-smpp**（二者均绑定队列 `sms_send_smpp` 时会竞争消费，易导致消息被错误消费者处理）。
- **仅调试旧 Python SMPP 路径**：`docker compose --profile legacy-smpp-python up -d`，且应**关闭或未部署** smpp-gateway，避免双消费者。
- 代码中 SMPP 在 Python `send_sms_task` 内已视为应由 Go 处理的路径，详见 `sms_worker` 内注释。
- **批量入队**：`batch_worker` / 小批量 API 对 SMPP 使用 `QueueManager.queue_sms_batch_smpp`，任务**直接**进 `sms_send_smpp`，**不再**先铺满 `sms_send` 再重投，降低双跳延迟。

### 3.1 smpp-gateway 环境变量（吞吐调优）

| 变量 | 默认值 | 含义 |
|------|--------|------|
| `SMPP_GATEWAY_WORKERS` | 32 | 并行处理 Rabbit `Delivery` 的 goroutine 数（与 DB/上游能力匹配调小或调大） |
| `SMPP_GATEWAY_PREFETCH` | 32 | Rabbit `basic.qos` prefetch，建议与 `WORKERS` 同量级；过大则未 ack 消息占用内存 |

回滚近似旧行为：可将二者设为 `1`。吞吐仍受 **`channels.concurrency`**（多 bind）与上游 SMSC 限制。

## 4. Compose Profile

| Profile | 作用 |
|---------|------|
| （默认） | 含 **prometheus**、**grafana**；不含 **worker-sms-smpp**（避免与 smpp-gateway 抢 `sms_send_smpp`） |
| `legacy-smpp-python` | 额外启动 **worker-sms-smpp**（仅在不使用 smpp-gateway 时使用） |

示例：

```bash
docker compose up -d

# 仅启用旧版 Python SMPP Worker（勿与 smpp-gateway 同开）
docker compose --profile legacy-smpp-python up -d
```

## 5. 扩容与约束

| 组件 | 扩容建议 | 约束 |
|------|----------|------|
| api | 可水平多副本（无状态前提） | 需共享同一 DB/Redis/RabbitMQ；注意限流与上传卷 |
| worker-sms | 按 `sms_send` 堆积增加副本 | 关注 DB 连接总数与 proxysql |
| worker | 按 `sms_dlr` / `data_tasks` / `celery` 堆积增加副本 | 同上 |
| smpp-gateway | 视上游 bind 与连接数评估 | 多副本时注意每副本均消费同一队列时的语义 |
| beat | **仅 1 个活跃实例** | 多实例会重复调度定时任务 |

## 6. 常见故障速查

| 现象 | 优先检查 |
|------|----------|
| 短信长期 queued | RabbitMQ、`worker-sms` 是否运行；`sms_send` 是否有 consumer |
| SMPP 不发送或随机失败 | 是否同时运行 **worker-sms-smpp** 与 **smpp-gateway**；Go 日志与 RabbitMQ 队列深度 |
| 长期停在「发送队列中」且 **`sms_send_smpp` 有消息、`consumers=0`** | **新版本 `smpp-gateway` 已对 RabbitMQ 断线自动重连消费**；若仍为旧镜像，可执行 `docker compose build smpp-gateway && docker compose up -d smpp-gateway`。历史环境亦可手动：`docker compose restart smpp-gateway` |
| DLR 不更新 | `worker` 是否消费 `sms_dlr`；Beat 是否运行；回调白名单与 Token |
| 定时任务不跑 | **beat** 单容器是否存活；`celery_app.beat_schedule` 是否变更 |

## 7. docker-proxy 安全边界

- **能力**：镜像 `tecnativa/docker-socket-proxy` 仅开放容器列表与 POST（如重启），不开放镜像/卷/执行等高危权限。  
- **风险**：若 `api` 被入侵，攻击面仍包含「对 Docker 的有限控制」。生产建议：内网隔离、强鉴权、审计；能不用则通过运维流水线替代。  
- **配置**：`api` 环境变量 `DOCKER_PROXY_URL` 指向 `http://docker-proxy:2375`。
