# 上线检查清单（架构优化相关）

发布含 **SMPP 部署约定**、**新 Celery 队列**、**RabbitMQ Prometheus 插件** 的版本时，请按顺序执行。适用于 Docker Compose 部署；其他编排可对照等价步骤。

## 1. 发布前（代码与配置）

- [ ] 生产 `.env` 已设置 **`APP_ENV=production`**（或团队约定值），**`APP_DEBUG=false`**。
- [ ] **`JWT_SECRET_KEY`**、**`MYSQL_*`**、**`RABBITMQ_*`** 为强口令，非仓库示例值。
- [ ] **`GRAFANA_ADMIN_PASSWORD`** 已设置（Grafana 默认不再使用硬编码 `admin`）。
- [ ] **`CORS_ORIGINS`**、DLR 相关 **`DLR_CALLBACK_*`** 与业务一致。
- [ ] 确认 **不会** 同时启用 **`smpp-gateway`** 与 **`--profile legacy-smpp-python`**（避免双消费者抢 `sms_send_smpp`）。

## 2. 发布动作（容器）

```bash
# 1）拉取最新代码 / 镜像后
docker compose pull   # 若使用预构建镜像
docker compose build api worker worker-sms worker-sms-smpp beat smpp-gateway worker-web bot frontend --parallel

# 2）滚动重启（至少必须重启 worker，以消费新队列）
docker compose up -d api worker worker-sms beat smpp-gateway
```

**必须重启 `worker` 的原因**：`worker` 的 `-Q` 已包含 **`webhook_tasks`**、**`integrations`**。若只更新代码不重启，Webhook 与 OKCC 任务会积压在 RabbitMQ 中无人消费。

可选：一并重启 `worker-sms`、`api`、`prometheus`，保证指标与路由一致。

## 3. RabbitMQ（插件与端口）

- [ ] 宿主机已存在 **`docker/rabbitmq/enabled_plugins`**（随仓库发布），Compose 已挂载。
- [ ] 首次升级后访问 **`http://<主机>:15692/metrics`**（或内网 `rabbitmq:15692`）确认有 Prometheus 指标。
- [ ] 若旧 **`rabbitmq_data`** 卷导致插件未加载：在低峰执行「启用插件」或按 [服务与队列矩阵.md](./服务与队列矩阵.md) 中说明评估是否重建卷（**会清空队列内未消费消息**）。

## 4. 发布后验证

- [ ] `curl -s https://<你的API域名>/health` 返回 `healthy`（或内网 `http://api:8000/health`）。
- [ ] RabbitMQ 管理台：队列 **`sms_send`**、**`sms_dlr`**、**`webhook_tasks`**、**`integrations`** 均有 consumer，且无异常堆积。
- [ ] 发一条 **HTTP 通道** 测试短信；若有 SMPP，发一条 **SMPP** 并确认 **`sms_send_smpp`** 仅由 **smpp-gateway** 消费（无 `worker-sms-smpp` 时也应成功）。
- [ ] **Beat** 仅 **一个** 容器在运行（多副本会重复调度定时任务）。

## 5. 回滚要点

若需回滚到旧版 Worker 队列列表（不含 `webhook_tasks` / `integrations`），须同时回滚 **`celery_app.py`** 与 **`docker-compose.yml`** 中 `worker` 的 `-Q`，否则会出现「任务进新队列、旧 Worker 不消费」的错配。
